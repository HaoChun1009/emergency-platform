> 版本:v2 · 初版 2026-05-29 · 最後更新 2026-06-01
> 目標:將「通報表單」擴充為「AI 驅動的智能通報系統」
> 定位:**Beta 原型**,驗證功能/流程/UI,並作為未來內網建置的規格範本

## 一、專案願景

打造一套以 AI 為核心的緊急事件通報系統,達成三大價值:

| 價值 | 說明 | 對應功能 | 目前狀態 |
|------|------|----------|----------|
| **加速** | 減少應變時間 | 智能輸入、自動分級、即時通知 | 輸入+分級✅ / 通知 ⏳ |
| **精準** | 智能分流、依規範判斷 | AI 三級分級、RAG 知識庫 | ✅ |
| **傳承** | 累積處理經驗 | RAG 知識庫、結案回流 | 知識庫✅ / 結案回流 ⏳ |

## 二、系統架構

### 目前實際架構(已完成)
```
使用者手機/瀏覽器(語音/定位/動態欄位/照片)
      │ 送出(可讀回應,取得 AI 結果)
      ▼
Google Apps Script 後端
      │     │            │
      │     │            └─→ RAG:讀取試算表「SOP知識庫」分頁
      │     └─→ Gemini API(文字+照片多模態)→ 三級分級 + 處置建議
      ▼
Google 試算表(資料儲存 + 管理後台)
照片 → Google 雲端硬碟「緊急通報照片」資料夾
      ▲
      └─ 完成畫面即時顯示 AI 研判(雙向回饋)
```

```

   ┌─────────────────────┐        ┌──────────────────┐
   │ AI 智能處理與分級 ✅  │◄──────►│ RAG 知識庫 ✅     │
   │ (Gemini,含照片辨識) │        │ SOP(分級/流程)  │
   └──────────┬──────────┘        └──────────────────┘
              │ 三級分級
       ┌──────┼───────┐
       ▼      ▼       ▼
   重大災害 廠級事故 一般事故
       │              │
```

## 三、功能模組詳解

### 1. 多元智能輸入 ✅
- **語音轉文字**:瀏覽器 Web Speech API,講話自動填入描述(需 HTTPS,GitHub Pages 已符合)。
- **自動定位 + 轉地址**:取得 GPS 經緯度,並用 OpenStreetMap 反向地理編碼成可讀地址;含高精度、逾時與明確中文錯誤提示。
- **動態災損欄位**:依事件類型顯示對應追問欄位(如火災→是否受困/煙霧狀況)。
- **現場照片**:可選相簿或拍照、多張;前端自動壓縮(最長邊 1280px、JPEG 70%)後上傳。

### 2. AI 智能處理與分級 ✅(核心)
- 後端於 `doPost` 呼叫 **Gemini API(`gemini-2.5-flash`)**,以 `responseSchema` 輸出結構化 JSON:
  - `severity`:三級分級(重大災害 / 廠級事故 / 一般事故)
  - `guidance`:依 SOP 的初步處置建議
- **照片辨識**:通報附照片時,一併送照片給 Gemini(多模態)結合文字判斷災情(最多 3 張)。
- **自動重試**:遇免費模型偶發 503/429/500 高負載時,漸進重試最多 3 次。
- 模型可日後切換(量大或需更強推理時改 Claude 等,只需改後端呼叫段)。

### 3. RAG 知識庫 ✅(簡化版)
- **做法**:依「事件類型」對應到該類 SOP → 注入提示詞 → AI 依公司實際 SOP 判級與建議。
- **儲存**:SOP 放在試算表「SOP知識庫」分頁(A欄=類型代碼,B欄=SOP內容),後端 `getKnowledge_()` 讀取。**不寫入公開程式碼**。
- **已實測**:火災通報能正確套用三級標準,並引用 SOP 條文(通報 119、疏散方向、各應變班職掌)。
- **擴充**:其他事件類型 SOP 依相同格式加入試算表即可,免改程式。
- **日後進階版**:文件量大時可升級向量資料庫(embeddings + 語意檢索),需更強後端。

### 4. 智能分流 ⏳(待做)
- 依 AI 三級分級決定路線:重大災害→即時通知;一般/廠級→標記狀態走原流程。
- 對應「免簽機制」:低風險、明確類型可設自動結案或免簽。

### 5. 即時通知 ⏳(待做,漸進升級)
| 階段 | 管道 | 說明 | 費用 |
|------|------|------|------|
| A | **Email** | Apps Script 內建 `MailApp` 直接寄信 | 免費 |
| B | **LINE 官方帳號** | LINE Messaging API 推播 | 免費額度內 |
| C | **電話語音** | Twilio 等撥打語音 | 付費,台灣門號設定較複雜 |

### 6. AI 雙向回饋 ✅
- 通報送出後,完成畫面即時顯示 AI 研判分級(三色徽章:紅/橘/綠)與處置建議,並附「僅供參考」聲明。

## 五、技術選型總覽

| 項目 | 選用 | 理由 |

## 七、風險與注意事項

1. **API 金鑰安全**:Gemini 金鑰存後端(Apps Script 指令碼屬性),**絕不寫進前端**。已落實。
2. **內部資訊/個資外洩**:公開 GitHub 只放前端程式;SOP 放試算表(不公開)、僅含判斷邏輯不含個資;原始 PDF 與 SOP 文字檔已 gitignore。⚠️ 日後任何內容上公開服務前須再次檢查。
3. **Apps Script / Gemini 配額**:免費版有上限,流量大時需評估遷移正式後端 / 付費 API。
4. **AI 判斷非 100% 正確**:分級為「輔助建議」,重大事件仍須人工複核;切勿讓 AI 全自動決定攸關生命安全的處置。
5. **公司網路阻擋**(見下節):內網無法連 Google 服務,Beta 以外部網路測試為主。
6. **可靠度**:緊急系統要求「不可漏接」,正式上線前需做失敗重試(已部分)、備援與壓力測試。

## 八、重要發現與專案定位

### ⚠️ 公司網路阻擋 Google 服務
- 實測:台泥公司網路下瀏覽器無法連 `script.google.com`(直開後端網址即失敗),送出出現 `Failed to fetch`。
- 手機行動網路 / 外部網路下系統完全正常。
- 結論:**系統本身無問題,純為公司網路政策阻擋外部 Google 服務。**

### 專案定位
- 本專案為 **Beta 原型**:驗證功能、流程、UI,作為規格範本。
- **未來由專業軟體人員接手部署於公司內網**,後端與網路阻擋問題屆時一併解決。
- Beta 階段沿用 GitHub Pages + Google 試算表(以外部網路測試)。
