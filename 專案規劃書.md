# 緊急通報平台 — AI 驅動系統規劃書

> 版本:v1 · 撰寫日期:2026-05-29
> 目標:將現有「通報表單」逐步擴充為「AI 驅動的智能通報系統」

---

## 一、專案願景

打造一套以 AI 為核心的緊急事件通報系統,達成三大價值:

| 價值 | 說明 | 對應功能 |
|------|------|----------|
| **加速** | 減少應變時間 | 智能輸入、自動分級、即時通知 |
| **精準** | 智能分流、免簽機制 | AI 分級、智能路由 |
| **傳承** | 累積處理經驗 | RAG 知識庫、結案回流 |

---

## 二、系統架構

### 目前狀態(已完成)
```
使用者手機/瀏覽器
      │
      ▼
通報表單(GitHub Pages 靜態網頁)
      │ 送出
      ▼
Google 試算表(Apps Script 後端)← 資料儲存 + 管理後台
```

### 目標架構
```
        多元智能輸入
   ┌─────────────────────┐
   │ 語音轉文字           │
   │ 自動定位 / 轉地址     │
   │ 災損樣態選擇          │
   └──────────┬──────────┘
              ▼
   ┌─────────────────────┐        ┌──────────────────┐
   │ AI 智能處理與分級     │◄──────►│ RAG 知識庫        │
   │ (Claude API)         │        │ ISO/SOP/結案經驗  │
   └──────────┬──────────┘        └──────────────────┘
              │
       ┌──────┴───────┐
       ▼              ▼
   重大事件        一般事件
       │              │
       ▼              ▼
   即時通知        原簽核路線
 (Email/LINE/語音)  (待簽核狀態)
              │
              ▼
   AI 雙向回饋(處置指引 + 補充建議)→ 回傳通報者
```

**重要架構決策**:前 3 階段沿用「GitHub Pages + Google Apps Script」,因為 Apps Script 能呼叫 Claude API、能寄 Email,免費且不用自建伺服器。等規模變大或需要更高可靠度時,再評估遷移到正式後端(Node.js / Cloud Run 等)。

## 三、功能模組詳解

### 1. 多元智能輸入
- **語音轉文字**:用瀏覽器內建 Web Speech API,按麥克風講話即自動填入「事件描述」。免費,但需在 HTTPS 環境(GitHub Pages 已符合)。
- **自動抓取定位**:已完成經緯度抓取;再加「反向地理編碼」把座標轉成可讀地址。
- **擴充災損樣態**:依事件類型動態顯示不同欄位(例如選「火災」時出現「是否有人受困」「煙霧狀況」)。

### 2. AI 智能處理與分級(核心)
- 通報送出後,後端把內容交給 **Claude API**,要求它輸出結構化結果:
  - `severity`:重大 / 一般
  - `category`:更細的事件分類
  - `reasoning`:判定理由
  - `guidance`:給通報者的即時處置指引
- 模型建議用 **Claude Haiku**(快、便宜)做分級;需要更強推理時用 Sonnet。

### 3. RAG 知識庫
- 整合 ISO 規範、SOP 作業流程、過往結案經驗。
- **漸進式做法**:
  - 第一版(簡單):把相關 SOP 文字直接放進 AI 提示詞(適合文件量少)。
  - 第二版(進階):文件切塊 → 產生向量(embeddings)→ 存向量資料庫 → 依語意檢索最相關片段給 AI。
- 向量資料庫選項:Supabase pgvector、Pinecone 等。

### 4. 智能分流
- 後端依 AI 分級結果決定路線:
  - **重大** → 觸發即時通知。
  - **一般** → 在試算表標記「待簽核」,走原流程。
- 對應圖中「免簽機制」:低風險、明確類型可設定為自動結案或免簽。

### 5. 即時通知(漸進升級)
| 階段 | 管道 | 說明 | 費用 |
|------|------|------|------|
| A | **Email** | Apps Script 內建 `MailApp` 直接寄信 | 免費 |
| B | **LINE 官方帳號** | LINE Messaging API 推播(LINE Notify 已停用) | 免費額度內 |
| C | **電話語音** | Twilio 等服務撥打語音 | 付費,台灣門號設定較複雜 |

### 6. AI 雙向回饋
- AI 回傳「建議處置指引」與「通報補充建議」,顯示在送出成功畫面給通報者,讓他知道當下該怎麼做。

## 七、風險與注意事項

1. **API 金鑰安全**:Claude 金鑰只能放後端(Apps Script Script Properties),**絕不可寫進前端網頁**,否則會外洩被盜用。
2. **Apps Script 配額**:免費版有每日執行時間/呼叫次數上限,流量大時需評估遷移正式後端。
3. **AI 判斷非 100% 正確**:分級結果應視為「輔助建議」,重大事件仍建議保留人工複核機制;切勿讓 AI 全自動決定攸關生命安全的處置。
4. **個資保護**:通報含姓名、電話、位置等個資,需注意儲存權限與法規遵循。
5. **可靠度**:緊急系統對「不可漏接」要求高,正式上線前需做失敗重試、備援與壓力測試。
6. **電話語音門檻**:Twilio 台灣門號與法規設定較複雜,建議最後再做。

- ✅ **AI 照片辨識**(Gemini 多模態):通報附照片時,一併送照片給 AI 結合文字判斷災情(最多 3 張)。已實測:描述模糊但附現場照片時,AI 能正確升級為「重大」
- ✅ 定位優化:加入高精度 GPS、逾時與明確中文錯誤提示(解決 iPhone 定位問題)
- ✅ **AI 雙向回饋**:通報送出後,完成畫面即時顯示 AI 研判分級(重大/一般,彩色徽章)與處置建議。前端送出改回可讀取回應模式以取得後端 AI 結果

### ⚠️ 重要發現:公司網路阻擋 Google Apps Script
- 測試確認:在**台泥公司網路**下,瀏覽器無法連到 `script.google.com`(直接開後端網址即失敗),導致通報送出出現 `Failed to fetch`。
- 但在**手機行動網路 / 外部網路**下,系統完全正常(送出、儲存、照片皆成功)。
- 結論:**系統本身無問題,純粹是公司網路政策阻擋外部 Google 服務。**

### 專案定位調整
- 本專案定位為 **Beta 原型**:用於驗證功能、流程與 UI/UX,並作為規格範本。
- **未來由專業軟體人員接手,部署於公司內網**,屆時後端與網路阻擋問題由內網建置一併解決。
- 因此 Beta 階段沿用 GitHub Pages + Google 試算表即可(以外部網路測試為主)。
